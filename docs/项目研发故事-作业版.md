# WeChat Moments AI 项目研发故事

## 一、项目从哪里开始

这个项目一开始并不是一个“大而全”的内容平台，而是一个非常明确的问题驱动型产品：我们想做一个专注于私域内容生产的 AI Agent，尤其是围绕“朋友圈文案生成”这件事，把从素材整理、内容生成、人工修改到最终发布前准备这条链路打通。

最早的设想其实更偏向“企业微信朋友圈智能营销助手”。当时的思路是，希望它能接入企业微信账号，读取历史朋友圈内容，自动学习账号风格，再结合产品资料和素材，生成新的朋友圈文案，并最终帮助用户完成审核和发布。也就是说，项目最早瞄准的是一个比较完整的“企业微信营销自动化工具”。

从产品想象上看，这个方向很完整，也很有商业感：既有账号管理，又有 AI 内容生产，还有审核发布闭环。但真正进入研发后，我们很快发现，一个项目的早期版本如果把“外部平台接入、自动发布、账号风控、内容生成”全部同时做进去，复杂度会非常高，尤其是对一个需要快速迭代、不断试错的 AI 产品来说，会拖慢最核心价值的验证。

所以这个项目后来的演化，实际上是一次非常典型的产品聚焦过程：我们逐步放弃了“平台连接和自动发布”这些重集成功能，把重心重新收回到“本地 AI 内容生产工作台”。

## 二、第一阶段：搭起一个能跑起来的全栈骨架

项目的第一个阶段，主要任务不是把功能做复杂，而是先把“工程底座”搭好。

当时选定的技术方案是：

- 前端使用 Next.js 14 + React + Tailwind CSS + shadcn/ui
- 后端直接使用 Next.js API Routes
- 数据库使用 PostgreSQL
- ORM 使用 Prisma
- AI 相关能力通过 LangChain.js 和模型配置模块组织

这一阶段的重点是把一个现代全栈项目最基本的结构建立起来，包括：

- 页面路由
- API 路由
- Prisma 数据模型
- 本地数据库连接
- 基础设置页与生成页

用更通俗的话说，这一阶段就像是在盖房子之前先把地基、承重墙和水电管线埋好。它不直接解决“文案写得好不好”的问题，但决定了后面的功能能不能顺利加进去。

## 三、第二阶段：从“账号工具”转向“本地内容工作台”

项目最重要的一次架构调整，发生在中期。

原来的设计里，有企业微信扫码登录、账号管理、自动发布、后台发布任务这些能力。但随着研发深入，我们发现这些功能虽然看起来很“完整”，却并不是产品最核心的价值所在。真正有价值、最值得深挖的，其实是“内容生产本身”。

于是我们做了一个比较大的重构决定：

1. 去掉企业微信 API 连接
2. 去掉自动发布和定时发布
3. 去掉对外部平台账号体系的依赖
4. 把整个产品改造成纯本地、本地数据库驱动的 AI 文案生产工具

这次重构之后，产品的角色发生了变化。

它不再是“帮你接管平台账号”的工具，而变成了“帮你把内容生产这件事做得又快又稳”的工作台。这样做有几个明显好处：

- 研发复杂度大幅降低
- 对外部平台规则变化不再那么敏感
- 用户不用先解决一堆接入问题才能开始用
- 产品可以更专注在“写什么、怎么写、怎么积累经验”这件事上

从产品思路上讲，这一步其实是把“自动化野心”收回来，换成“先把最有价值的内核做好”。

## 四、第三阶段：知识库、内容形式、产品与素材的结构化

当产品聚焦到内容生产之后，下一个问题就变成了：AI 到底应该吃什么数据？

一开始，系统里有“风格”这个概念，意思是让 AI 学习某种文案气质，比如温柔、专业、亲切等。但在不断讨论和使用之后，我们发现，真正更关键的不是抽象风格，而是“内容形式”和“具体产品”的区别。

于是知识库的结构被重新整理成更贴近真实运营工作的样子：

- 内容形式
- 产品
- 产品下的图文素材

比如同样是品牌内容，不同内容形式可能是：

- 情侣互动种草
- 友情深聊卡牌
- 女性视角观点表达
- 青春治愈与毕业季

而在每种内容形式下面，又对应具体产品，再往下才是该产品历史上积累下来的文案素材和图片素材。

这一步的意义很大，因为它让系统不再只是“收集一堆素材”，而是开始具备内容生产的组织能力。AI 在生成文案时，不是漫无目的地去翻素材，而是先知道自己正在处理哪一类表达、哪一个产品，再去拿相应的图文记忆。

## 五、第四阶段：从“写一条文案”进化成“可反复迭代的内容工厂”

知识库结构建立起来之后，项目开始进入真正的内容生成阶段。

这一阶段逐渐形成了现在的内容工厂逻辑：

1. 选择平台：朋友圈或小红书
2. 选择内容形式
3. 选择具体产品
4. 输入本次文案目标
5. 系统读取本地素材、产品关键词、图片分析结果和平台记忆
6. AI 生成初稿
7. 用户继续修改、再生成、定稿

这里面有一个很重要的设计原则：不是让 AI 一次性写出“最终答案”，而是把它当成一个能快速产出第一版、再陪用户迭代的协作者。

所以系统后来又加上了：

- 多版本内容管理
- AI 改写
- 手动修改
- 最终定稿
- 手工发布日历记录

这意味着项目已经不再只是“一个生成按钮”，而更像一个本地内容生产工作流。

## 六、第五阶段：让小红书成为独立的研究与生成模块

随着产品不断细化，我们意识到，朋友圈和小红书虽然都属于内容平台，但它们的写法、节奏、用户预期其实差异很大。

如果把两边的经验混在一起，最后很容易变成“哪里都不像”。所以项目后来把小红书独立拉出来，形成了一套自己的研究与生成闭环。

这套闭环包括两部分：

### 1. 小红书研究中心

这里不是直接写文案，而是先研究：

- 抓取某个主题下的小红书帖子
- 导入本地研究库
- 对帖子进行清洗、筛选、删掉不想保留的样本
- 基于剩余样本重新生成研究洞察

### 2. 小红书内容生成

当用户在内容工厂里选择小红书平台时，系统会自动读取：

- 本地文案记忆
- 产品信息
- 图片分析结果
- 已生成的小红书研究洞察

也就是说，小红书部分不是单独存在的一页“研究报告”，而是反过来服务于文案生成本身。

## 七、第六阶段：处理“抓取串题”和“任务管理”这些真实问题

项目发展到这一步后，遇到的问题就不再是“有没有功能”，而是“功能在真实使用中稳不稳”。

比如我们遇到过一个非常真实的问题：明明抓的是新的小红书话题，系统却把以前抓到的“毕业季”内容混了进来。后来排查发现，问题并不是用户操作错了，而是底层导入逻辑有漏洞：

- MediaCrawler 的 JSON 文件按天追加
- 同一天的多个搜索主题会写进同一个文件
- 系统如果不按 `source_keyword` 做严格隔离，就会把不同主题的帖子混导进来

这类问题非常典型，也很有价值，因为它说明项目已经进入了“要经得起真实使用”的阶段。后来我们做了几层修复：

- 抓取导入时按 `source_keyword` 严格过滤
- 同名研究集合重新导入时清空旧样本和旧洞察
- 内容生成读取研究洞察时，只允许主题相关的洞察参与生成

类似的，还有抓取任务删除问题、运行中任务误删保护、历史任务和研究集合之间的关系清理等。这些工作虽然没有“新功能”那么显眼，但实际上是产品成熟过程里非常关键的一部分。

## 八、项目现在是什么样子

发展到现在，这个项目已经从最初的“企业微信朋友圈智能营销助手”演化成了一个更聚焦、更稳定、也更适合继续打磨的本地 AI 内容工作台。

它目前的核心能力是：

- 按品牌项目隔离数据
- 支持朋友圈和小红书两个平台
- 用“内容形式 -> 产品 -> 图文素材”的方式组织知识
- 支持本地文案记忆导入
- 支持图片分析辅助写作
- 支持小红书主题抓取与研究
- 支持人工筛选研究样本、删帖、重生洞察
- 支持多版本文案生成、改写与定稿

更重要的是，它已经不只是一个“会写文案的页面”，而是逐步形成了一个完整的内容生产工作流：

研究 -> 清洗 -> 洞察 -> 生成 -> 修改 -> 定稿 -> 发布记录

## 九、这个项目体现了怎样的研发思路

如果把整个过程总结成一句话，这个项目并不是按照“先设计最宏大的版本，再一次性做完”的路线发展的，而是沿着“先搭骨架 -> 找到最核心价值 -> 不断收缩复杂度 -> 让工作流越来越顺手”的方式推进的。

它体现了几种比较典型、也比较成熟的研发思路：

### 1. 先有方向，再不断聚焦

项目最开始目标很大，但后来主动砍掉了不属于核心价值的部分，把精力集中在最值得做深的内容生产上。

### 2. 让 AI 成为工作流的一部分，而不是孤立功能

AI 不是只负责“写一句话”，而是被放进素材管理、研究学习、生成改写、定稿等整条链路中。

### 3. 不只做“能用”，还要做“能长期用”

后期大量精力花在样本清洗、串题修复、删除逻辑、任务状态保护等问题上，这说明项目已经从“功能演示”迈向“真实使用”。

### 4. 产品化不只是加功能，而是降低理解成本

后来很多优化，其实都是在减少用户的学习负担，比如把研究结果直接接进生成，把样本筛选放在集合卡片内，把任务、集合、洞察之间的关系做得更清楚。

## 十、接下来还能往哪里发展

如果继续往前做，这个项目已经有几条很自然的方向：

- 让研究洞察在内容生成页和任务详情页里更透明可见
- 做更完整的“研究 -> 生成”一键工作流
- 强化不同品牌项目之间的隔离与切换体验
- 继续提高小红书研究样本的筛选效率
- 把内容质量评估和复盘能力做得更强

但无论往哪个方向继续，它现在最关键的基座已经立住了：它不是单点功能，而是一套围绕内容生产的本地智能工作系统。
